# jetstream-fs

This file-system project is a file system for AI, RAG, and robotics model data. It is designed to be robust, reliable, scalable, fast, and easy to use. The file is organized into 4 KB blocks and operates on a single block file or device with a fixed size of const BLOCK_SIZE = 4 KB. It does not rely on any single point of failure such as a header block or large index blocks. The system starts at OFFSET = 0x1000 and has a total file system size of const FILE_SIZE = 100 MB.

When calculating block locations, the modulo operator is used to compute the block index from the offset and size. The default file path is PATH=/mnt/rag.dat, with a total size of OFFSET + FILE_SIZE.

Blocks are never addressed directly by index. Instead, the index is always derived from a user-provided locator. The locator is always a SHA‑256 value, derived from the hash of the content, the hash of a string, or the hash of a random value. To locate a block, we compute OFFSET + M(locator) * BLOCK_SIZE, where M(locator) = locator modulo FILE_SIZE. If the block is occupied, we retry using OFFSET + M(sha256(locator)) * BLOCK_SIZE. If that block is also busy, the same algorithm is repeated up to DEPTH=5 levels deep.

Each block contains both content and an index. The index may be simple, such as a hash of the content or a hash of a path string, or more complex, such as an embedding vector or an AI model’s interpretation of the block content. This allows the system to store complex data, including raw training data, directory tree indexes, hash indexes, RAG embedding indexes, diff trees, and AI model blocks.

Empty blocks are represented as all-zero bytes to support NAND flash. The data block format is written in simple English. For example: “This is a raw data block of 0x300 bytes at offset 0x200. It is identified by random hash <sha256>.” Random hashes may be cryptographically secure or deterministic, such as sha256('/var/log/messages'). Another pattern is: “This is a raw data block of 0x300 bytes at offset 0x200. It is identified by content hash <sha256-content>.” The system verifies the content hash when saving or loading. Another pattern is: “This is a raw data block of 0x300 bytes at offset 0x200. It is identified by embedding vector [0.1, 0.2, 0.3, ...].” These index types are non-exclusive and may appear as a newline-separated list.

This structure can be used by language models to retrieve and update data. It can be used by real-time robotics automation to compute torques and voltages deterministically and within constant time. It can store models and training data, enabling queries such as “give me the evidence” or “fact check this answer.” It can support AI training workflows by refining and improving models on the fly or storing base models alongside fine‑tuned versions. It can hold training data, knowledge graphs, and model blocks. It can store knowledge graphs composed of abstracted, hash-referenced connections between data blocks. It can maintain a tree of changes similar to Git, supporting multiple versions of the same model through chains of block hashes. Because it relies only on block locality, it is well-suited for extreme distributed storage environments such as Kove or UnifabriX.

Threading is simplified by enforcing constant retrieval time per block. The system is divided into PARTITION_COUNT equally sized partitions, each protected by its own mutex. When a thread accesses a block, it acquires the mutex for that partition and then calls the appropriate function using that partition’s offset and size. Since block searching occurs entirely within the partition, all deeper searches operate on one fourth of the total size, and hash collisions resolve to locations within the same partition.

The mutex algorithm ensures ACID compliance at the single-block level, maximizing parallelism. Multi-block structures require their own application-level synchronization mechanisms, such as versioning, journaling, snapshot isolation, or eventual consistency, consistent with the CAP theorem. Although this mutex approach may seem slow, it is highly flexible. It allows the number of partitions to match the expected number of concurrent threads, ensuring locking occurs only when necessary. Partitions typically sit behind a NUMA node or PCIe bus with limited bandwidth and IOPS, making finer-grained synchronization unnecessary.
